# CONSENSUS_云开发迁移到Node后端

## 一、最终需求描述

### 1.1 目标

将当前儿童视力管理项目从“微信云开发 + 云函数 + 云数据库 + 云存储”迁移为“Node 后端 + MySQL + 本地文件存储”，并确保小程序端与管理后台端都能在不依赖微信云开发业务能力的前提下正常运行。

### 1.2 交付范围

- 新建并维护根目录统一文档体系
- 收敛 `node后端（新增）` 为唯一业务后端
- 完成移动端 `/api/v1/mobile/*` 接口
- 完成后台端 `/api/v1/admin/*` 接口
- 完成 MySQL 数据库初始化脚本与表结构
- 完成本地文件上传与静态访问
- 完成小程序与后台接口切换
- 完成联调、验收与文档更新

### 1.3 端范围

| 端 | 保留/调整 | 说明 |
| --- | --- | --- |
| 小程序端 | 保留 | 仅替换后端调用方式与图片访问方式 |
| 管理后台 | 保留 | 仅替换后端调用方式与图片上传/预览方式 |
| Node 后端 | 新后端主实现 | 成为唯一业务后端 |
| 云函数目录 | 仅历史参考 | 不再承接业务新改动 |

## 二、验收标准

### 2.1 文档验收标准

- 已创建 `说明文档.md`
- 已创建 `ALIGNMENT`、`CONSENSUS`、`DESIGN`、`TASK`
- 文档中包含项目结构现状、目标结构、API 清单、MySQL 规划、五阶段任务拆分
- 每份文档内容可直接作为后续开发输入

### 2.2 业务验收标准

- 小程序端可完成手机号登录
- 小程序端可完成微信快捷登录
- 小程序端可完成孩子档案获取/新增/修改/删除
- 小程序端可完成预约项目查询、排班查询、预约提交、预约记录查询
- 小程序端可完成检测记录查询、详情、创建、更新
- 小程序端首页可正常读取轮播图与协议内容
- 管理后台可完成管理员登录
- 管理后台可完成用户、孩子、学校班级、轮播图、预约项目、预约排班、预约记录、检测记录、协议与隐私的 CRUD 或管理动作
- 管理后台统计面板可展示基础统计数据
- 全项目运行过程中不再依赖云函数与 CloudBase 业务接口

## 三、技术实现共识

### 3.1 前端共识

- 小程序端统一新增 HTTP 请求封装，替换 `wx.cloud.callFunction`
- 管理后台统一使用 REST API，替换 `cloudbasePost`
- 图片展示统一改为基于后端静态 URL 的访问方式
- 尽量保持现有页面、交互和字段命名不变，控制改造成本

### 3.2 后端共识

- 后端框架采用现有 `node后端（新增）` 中的 Express 骨架
- 路由分为 `/api/v1/mobile` 与 `/api/v1/admin`
- 认证采用 JWT 双端隔离
- 文件存储首版采用本地目录 + 静态资源访问
- 微信快捷登录通过微信官方 `jscode2session` 接口实现
- 后端不再使用 `admin_sessions` 表做临时 token 会话，而改为 JWT

### 3.3 数据库共识

- 数据库采用 MySQL
- 数据表优先对齐原云集合命名与字段语义
- 嵌套结构字段（如验光数据、诊断数据）首版允许使用 JSON 字段
- 所有时间字段统一保留 `created_at`、`updated_at`
- 首版不做历史数据迁移，不编写云数据导入脚本

## 四、边界与限制

### 4.1 范围限制

- 只迁移当前已存在业务
- 不新增支付、消息订阅、复杂审核流程等新链路
- 不构建生产级文件对象存储与 CDN 能力
- 不补历史云数据

### 4.2 不确定项处理规则

- 若实现中遇到“是否改变业务规则”的问题，优先保持与现有云函数逻辑一致
- 若遇到可选增强能力，优先保证核心功能跑通，再做增强
- 若遇到需要用户决策的业务变化，先暂停并记录在任务文档中

## 五、最终共识结论

本次实施以“先完成文档，后完成 Node 后端与前端切换”为主线，目标是使 `node后端（新增）` 成为唯一业务后端，小程序和管理后台全部通过 HTTP API 工作，微信云开发只保留为历史参考代码。

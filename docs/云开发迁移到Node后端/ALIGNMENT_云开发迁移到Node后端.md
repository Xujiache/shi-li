# ALIGNMENT_云开发迁移到Node后端

## 一、原始需求

### 1.1 用户原始诉求

1. 先熟悉当前项目。
2. 先搭建项目文档，再整理项目结构。
3. 将当前微信云开发相关后端能力整体迁移到 `node后端（新增）`。
4. 不想继续使用微信云开发。
5. 本地使用 MySQL 作为数据库。

### 1.2 已提供材料

- 当前完整项目代码
- `小程序端/最终（视力）/cloudfunctions/*`
- `后台/art-lnb-master/*`
- `node后端（新增）/*`
- 用户确认的迁移决策：
  - 小程序端与管理后台都切换到新后端
  - 微信云开发中的旧数据与旧文件都不保留
  - 保留“手机号登录 + 微信快捷登录”

## 二、需求理解

### 2.1 项目目标理解

当前项目的业务已经完整，但后端能力依赖微信云开发。用户希望将后端统一迁移到常规 Node 后端，以便脱离云函数和云数据库，使用本地 MySQL 自主维护数据与服务能力。

### 2.2 业务角色理解

| 角色 | 说明 | 主要诉求 |
| --- | --- | --- |
| 家长/普通用户 | 使用微信小程序登录与维护孩子信息 | 登录、档案、预约、查看检测记录 |
| 管理员 | 使用管理后台维护业务数据 | 管理用户、孩子、预约、轮播图、协议、统计 |
| 系统服务 | Node 后端 | 承接认证、业务、数据存储、文件上传 |

### 2.3 多端范围理解

| 端 | 当前实现 | 迁移目标 |
| --- | --- | --- |
| 小程序端 | `wx.cloud.callFunction` + 云存储 | HTTP API + Node 本地文件服务 |
| 管理后台 | CloudBase HTTP 网关 + CloudBase 存储 | REST API + Node 本地文件服务 |
| 后端 | 云函数 + 云数据库 | Express + MySQL + JWT |

### 2.4 核心业务流程理解

1. 用户通过手机号登录或微信快捷登录进入小程序。
2. 用户选择或维护孩子档案。
3. 用户查看首页轮播、协议内容、预约项目和检测记录。
4. 用户发起预约并查看历史预约。
5. 管理员通过后台管理用户、孩子、轮播图、预约、检测记录、协议和统计数据。

## 三、任务边界

### 3.1 本次纳入范围

- 迁移云函数业务逻辑到 Node 后端
- 设计并落地 MySQL 表结构
- 替换小程序端云函数调用
- 替换管理后台 CloudBase 网关与 CloudBase 存储依赖
- 统一文件上传与静态访问方案
- 补齐迁移期文档体系

### 3.2 本次不纳入范围

- 迁移历史云开发数据
- 迁移历史云存储文件
- 新增全新业务模块
- 复杂多角色 RBAC 和审计体系
- 正式生产部署脚本、容器化和 CI/CD

### 3.3 约束条件

- 需要保留原有业务语义，降低前端改造范围。
- MySQL 账号密码不得写入源码和文档正文。
- 需同时兼容小程序端和管理后台两套前端。
- 微信快捷登录不能继续依赖云开发，需要改为官方登录链路。

## 四、现有项目对齐

### 4.1 已有文档与模式

- 根目录当前缺少统一的 `说明文档.md`。
- `小程序端/最终（视力）/docs` 下已有多份需求/设计/任务文档，可复用其术语和写作风格。
- `后台/art-lnb-master/docs/vision-admin` 下已有本地对接文档，可复用管理后台接口命名习惯。

### 4.2 可复用结构

| 位置 | 可复用内容 | 迁移方式 |
| --- | --- | --- |
| `node后端（新增）/routes` | Express 路由总入口 | 保留并扩展 |
| `node后端（新增）/utils/db.js` | MySQL 连接池 | 保留并增强事务能力 |
| `node后端（新增）/utils/jwt.js` | 双端 JWT | 保留并补充用户信息载荷 |
| `小程序端/最终（视力）/cloudfunctions` | 现有业务规则 | 迁移为 Node 服务与 SQL |
| `后台/art-lnb-master/src/api/vision-admin.ts` | 管理后台接口封装入口 | 改造为 REST API 调用 |

## 五、疑问澄清

### 5.1 必须确认项

| 优先级 | 问题 | 影响范围 | 当前状态 |
| --- | --- | --- | --- |
| 高 | 是否小程序端和管理后台都立即切换到新后端 | 所有前端与后端范围 | 已确认 |
| 高 | 是否保留旧云开发数据/文件 | 数据迁移策略、文件策略 | 已确认：不保留 |
| 高 | 新后端保留哪种登录模式 | 用户表、微信登录接口 | 已确认：手机号 + 微信快捷登录 |

### 5.2 可带假设继续项

| 问题 | 默认假设 | 风险 |
| --- | --- | --- |
| 文件存储首版选型 | 使用 Node 本地文件目录 | 后续部署到多机时需升级对象存储 |
| Redis 是否首版启用 | 可用但不作为首版阻塞条件 | 若未启用，仅影响缓存与扩展能力 |
| 后台权限模型 | 继续保留 `users.is_admin` 简化模式 | 后期若扩角色系统需补充角色表与权限表 |

## 六、当前对齐结论

本次迁移的本质不是“新增一个旁路后端”，而是将现有云函数后端正式替换为 `node后端（新增）`，并使其成为唯一业务后端实现。迁移目标明确、边界清晰，可直接进入共识与设计阶段。

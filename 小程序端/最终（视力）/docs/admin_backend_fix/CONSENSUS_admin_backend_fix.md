# CONSENSUS_admin_backend_fix

## 1. 最终需求描述

在不大改现有架构（微信云开发 + 外部后台调用云函数）的前提下，完成两类修复：

1) **小程序多账号不串号**：同一微信（同 OPENID）下切换不同手机号账号时，业务数据（孩子档案、预约记录、检测记录等）按账号隔离；同时修复 `user_manager` 在 `user_id` 场景下返回/更新 `_id` 可能为 `undefined` 的问题。

2) **后台管理能力完善**：后台除了用户管理外，补齐协议与隐私内容的可配置能力（`system_config.key=terms_and_privacy`），并提供对应页面。

## 2. 验收标准（可执行）

### 2.1 小程序（功能+隔离）

- **账号隔离**
  - A 手机号账号登录后创建的孩子档案/预约/记录，B 手机号账号登录后不可见（同一微信 OPENID 下也不可见）。
- **身份解析正确**
  - 小程序任意业务云函数请求都能携带 `user_id`（来自 `current_user_id`），云函数按 `user_id -> users.phone` 进行归属查询。
- **微信登录“默认账号”可预期**
  - 同一 OPENID 下存在多个 `users` 记录时，微信登录默认进入最近登录过的账号（按 `last_login_at/updated_at/created_at`）。
- **安全加固**
  - `profile_manager.update` 更新孩子档案时，若孩子不属于当前账号（`parent_phone` 不匹配），返回失败且不会写库。

### 2.2 后台（可配置 + 可保存）

- 后台侧边栏可进入 `儿童视力管理 > 协议与隐私`。
- 保存成功后，再次进入页面能读取到最新保存内容。
- 小程序登录页协议内容可加载到最新配置（优先走云函数 `data_manager.terms_get`）。

## 3. 技术实现共识

- **小程序侧身份主键**：以 `user_id`（`users._id`）作为强身份；云函数内部通过 `user_id` 找到 `users.phone`，以手机号归属作为数据隔离依据。
- **OPENID 回退策略**：仅在缺少 `user_id` 时使用；在同一 OPENID 下多账号时优先“最近登录账号”，避免随机命中旧账号。
- **后台鉴权**：保持现有 `admin_sessions.token` 机制；后台前端通过 `withToken()` 将 token 透传给 `admin_manager`。
- **系统配置**：协议与隐私存储在 `system_config`，key 固定为 `terms_and_privacy`；后台编辑写入，C 端只读获取。

## 4. 边界限制

- 小程序 phone 登录仍为轻量实现（未引入完整 JWT/会话系统）；本次不扩展到强安全的“每次请求带签名 token”。
- 不做历史数据的强制迁移（仅通过逻辑修正降低串号与误写风险）。
